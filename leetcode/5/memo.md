# Step 1

## `step1_brute_force.cpp`

動的計画法などが頭をよぎったが、上手い方法が思いつかない。
C++ なら 1 秒間で `10^8` (`~10^9`) 回くらいの処理が行えるので、

> `1 <= s.length <= 1000`

という制約から、部分文字列を決める start と end を全通り試して (かなり大雑把に) `10^6`, 回文チェックで大体 `10^3`, 合わせて `10^9` 回くらいの処理が必要になりそうだが... 枝狩りをすれば 1 秒くらいで終わるか？ -> `step1_brute_force.cpp` で全てのテストケースをパスできた。

時間計算量: `O(n^3)` (`部分文字列のstart/endの組 * 回文チェック`)、空間計算量: `O(n)` (最悪入力文字列と同じ長さのものを作って出力しなければならない。)

だが、これは効率の良いやり方ではないだろうな。

# Step 2

## `step2_dp.py`

二次元配列で各 (start, end) のペアが palindrome かどうかを保持すれば、回文チェックが O(1) でできそうなのはわかったが、それをどう埋めていけば良いのかわからなかった。
ChatGPTと数回やりとりしたらわかって、私の理解は以下の通り。

`s[start:end + 1]` が 回文になるのは、`s[start] == s[end]` かつ `s[start + 1 : end]` が回文の時 (簡単のため `end - start + 1 > 2` の時のみ着目)。
(start, end) の組が回文かどうか記録、再利用できればいいのだが、(start, end) をみるときに (start + 1, end - 1) の結果が必要なので、
start は大きい方から小さい方に減少させ、end は小さい方から大きい方に増加させて表を埋めていく必要がある。

なんとなくわかったような気がするが、これを自然と思えるまでにはまだ時間がかかりそう。

## `step2_manacher.py`

[Manacher's algorithm](https://en.wikipedia.org/wiki/Longest_palindromic_substring) というものがあるらしい。

Wikipediaに載っている疑似コードを見ながら、自分の理解をコメントに加えてPythonで書いたら `step2_manacher.py` になった。

かなり読解に苦労したし、完璧に再現できるようになるまでかなり時間がかかりそうなので、一旦、そういうアルゴリズムもあるんだくらいに留めておくことにする。

LeetCodeの実行時間はあまりアテにならないが、何回か走らせたら明らかにDPの解法よりも速そうだ。

時間計算量 O(n), center は単調増加し O(n) + radius を広げる操作も全体を通して O(n) 程度しか起こらないため。(まだしっくりはきていない。)
空間計算量 O(n), separators を挿入して 2n + 1 の長さの文字列を作成 O(n) + 長さ 2n + 1 の `palindrome_radii` を保持 (n)
