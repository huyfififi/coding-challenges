# Step 1

coinsをソートしてループを早めに抜ける方法も記憶の片隅にあったのだが、そんなに嬉しさを感じなかったのでスルーした。
コインの最大枚数が高々10枚くらい、外側のループを最大10^4回くらい回すので、処理は10^5回くらい。
C++では大雑把に10^8\~9個ほど命令を一秒間で実行できるので、雑にループを10^8/秒処理できると置くと 10^5 / 10^8 = 10^(-3) s = 1 msくらいのオーダーの実行時間だと予想される。
よしんばcoinのループを合計で半分にできたとしても、影響はかなり少ないように見えるので、ソートする一手間を加えることに嬉しさを感じなかった。

step 1を実装後、考え直してみたら、coinの最大値が2^31 - 1なのに対して、amountが最大でも10^4なので、coinの大半/全てがamountを超えている可能性もあるのか、と気づいた。
そうすると、全てのcoinのループが初回で終わるので、coin由来のループが12 -> 1に減らせて、(ループ由来の) 実行時間が 約1/10 になるのは少し嬉しい場合もあるかもな、と思った。
ただ、この辺は制約に依存していて、coinが小さくamountが大きい、例えばcoins = [1, 5, 10], amount = 10^20 のような時にはソート + break がほぼ影響しないだろうと思うので、今回の問題設定・bottom upのDPでは気にしてソートする必要はないのではと考える。
